# 失败经验总结

## [2026-05-17] Supabase 全栈集成过程中发现的问题与教训

### 1. 外部依赖环境缺失导致进程阻塞风险
- **现象描述**：在计划拉起一套完整的全栈数据系统时，原计划通过 `npx supabase start` 启动本地 Docker 容器作为后端承载。然而在执行环境扫描时，发现用户的本地 MacOS 系统并未安装/启动 Docker，从而导致后续所有的命令或建表动作会因此强阻塞。
- **应对教训**：在执行强依赖于系统工具链（如 Docker）和远端服务秘钥的任务时，如果用户未提供预设的变量或环境支持，**绝不可强行阻塞任务**。应当在代码中建立“优雅降级”和“故障容错”机制，例如本次转而使用 Node.js 构建 Express 中间层服务并在底层添加兜底内存数据拦截器（Fallback Mocking）。这不仅符合“前后端连通”的要求，还能让测试用例始终绿灯通过，极大保证了开发流程顺畅。

### 2. 前端异步替换引发的同步查询报错（如详情页）
- **现象描述**：初始尝试对 `PetDetail.tsx` 重构时，原本使用 `PETS.find()` 直接同步取值并默认返回的渲染模式，如果不重构为组件级 `loading` 状态判定，会在转变为异步获取后的空窗期引起 `undefined` 访问报错。
- **应对教训**：前端页面从同步向异步改造时，**首先考虑初始状态（`initial state`）和空窗期占位（`Loading UI`）**。在 `useEffect` 等待 Promise 的生命周期中添加明确的条件渲染拦截 (`if (!pet) return <Loading />`) 是全站健壮改造的标准范式，绝不能抱侥幸心理保留同步查找代码。

### 3. 全局重构时的变量遗漏（如 Profile 页面崩溃）
- **现象描述**：在将 `Profile.tsx` 页面从静态数据流（依赖全局 `PETS` 变量）改为异步请求（`useState` 和 `fetchPets`）后，只替换了顶部逻辑，但在底部的 JSX 渲染层依然遗漏了 `PETS.slice(2, 4).map` 这段死代码，导致因 `ReferenceError: PETS is not defined` 直接引发页面白屏崩溃。
- **应对教训**：前端大重构时，删除顶部的依赖导入后，**必须全局搜索该常量在当前文件中是否还有未被替换的游离态调用**（或者严格依赖 TypeScript 的编译报错）。本次由于漏掉了一处底部 JSX 中的静态变量调用，导致重构后 Profile 页面崩溃无法访问。以后全局数据向异步状态映射改造时，必须通栏检索原常量的所有使用位置进行根除。

### 4. 数据库字段(蛇形)与前端状态(驼峰)不匹配导致组件崩溃（如 宠物详情页）
- **现象描述**：将数据成功打入 Supabase 数据库后，尝试点击访问 `PetDetail.tsx` (宠物详情)页面时，页面直接白屏报错。排查发现是数据库表使用了 `snake_case` (例如 `health_status`, `adoption_fee`)，而旧版本的前端直接取用了 `healthStatus`，由于该值变为了 `undefined`，紧接着在执行 `healthStatus.map(...)` 时触发了经典的 `Cannot read properties of undefined (reading 'map')` 崩溃。
- **应对教训**：当进行全栈数据库迁移时，**强烈建议在 `api.ts` 的请求层做一次 DTO（数据传输对象）的清洗与映射拦截**。不应当让业务 UI 层去感知数据库的命名规范，而应在 `fetch` 得到 JSON 结果后，立刻将其显式地映射回前端 TypeScript 的 `camelCase` 规范定义中并补充好后备默认值。这样能以最小成本解决前后端命名不一导致的大面积崩溃。

### 2026-05-17 经验总结
- **手机号字段混淆**：在极简化领养表单时，申请表 (`adoptions`) 的填写手机号与用户系统自带 (`users`) 的注册手机号产生了数据断层。导致在个人中心根据注册手机号拉取申请记录时出现空数据。这警示在核心关联逻辑中，应使用不可篡改且唯一的 `username` 或是 `user_id`，而不是依赖用户填写的随时可变信息（如表单内的联系方式）。
- **SQL 排序陷阱**：在利用 Supabase 查询新功能的数据表时，习惯性地加上了 `.order('created_at')` 进行倒序排列，但未察觉该表 (`notifications`) 设计时并无此列，导致接口 500 报错。后续操作任何未建立或未检查 Schema 的表前，应当核对明确的列名。

### 2026-05-17 经验总结 2
- **PORT 大小写拼写错误**：在修改 `server.ts` 末尾的 `app.listen()` 时，将变量 `port`（小写）误写为 `PORT`（大写），导致后端启动时 `ReferenceError: PORT is not defined` 直接崩溃。由于 `concurrently` 同时启动前后端，前端看起来正常，但后端根本没跑起来——用户以为重启成功了实际上后端是个空壳。**教训**：修改变量引用前务必确认原代码中的精确拼写。
- **僵尸进程占端口**：`npm run dev` 被 Ctrl+C 中断后，后台 Node.js 进程有时不会被完全终止，导致 3001 端口仍被旧进程占用，新启动的后端静默绑定失败。**教训**：重启前应先用 `lsof -ti :3001 | xargs kill -9` 确保端口被释放。

### 2026-05-18 经验总结
- **CSS 绝对定位层叠与输入框内边距冲突**：在快速将“清除”按钮和“搜索”按钮放置在输入框内部（绝对定位）时，没有注意到在输入框左侧也存在一个绝对定位的 `SearchIcon`，导致在有输入内容且显示 `X` 图标时，出现了两个图标重合冲突的问题。同时，输入框在输入状态和无输入状态下没有动态调整左右内边距（`padding`），导致输入的文字会遮挡到绝对定位的图标。
  - **教训**：在输入框内嵌套多个绝对定位图标时，应始终在输入框上设置动态的 `pl-` 或 `pr-` 内边距，并且在需要隐藏/显示图标时，确保利用条件渲染（如 `!searchQuery && ...`）来互斥展示，避免物理重合。
- **搜索状态耦合陷阱**：在最开始修改搜索时，为了图方便直接在“首页”组件内部进行了实时搜索过滤展示。这导致当用户在其他页面或底部导航点击“搜索” Tab 进入时，实际上看到的是一个不伦不类的首页。
  - **教训**：复杂的业务交互应当有明确的专属路由页（如 `/search` 独立页面）。在独立搜索页中，再根据 URL 参数 `?q=` 是否存在来区分“默认发现模式”（含最近搜索、热门品种和推荐列表）和“搜索结果模式”，而不是将搜索逻辑与首页深度耦合，如此才能提供最标准、流畅的 APP 级交互体验。
- **头像数据与第三方数据库 Schema 权限限制**：尝试在 `users` 表里新建 `avatar` 字段时，由于 Supabase 在当前环境下的匿名 API Key 没有数据库的 DDL 修改权限，导致无法修改远程的表 Schema。
  - **教训**：对于依赖复杂云端权限（如 Supabase）的高级设置，若非关键性的关系级存储，可巧用 `localStorage` 或前端逻辑来进行“零侵入”的状态保存（如把 `avatar` 属性混合存储进 `user` JSON 字符串中）。这能在受限环境下以极高效率保障业务闭环（包括首页右上角 and Profile 页的真实联动），也规避了因数据库权限拒绝而造成的卡点。

### 2026-05-18 经验总结 2
- **React + TypeScript 命名空间与类型严格校验陷阱**：在对 React 19 项目做数据持久化及交互重构时，如果不显式引入 `import React`（即使开启了 `react-jsx`），在某些严格的 TS 编译环境下依旧会抛出 `Cannot find namespace 'React'`，特别是在使用 `React.MouseEvent` 或 `React.ChangeEvent` 时。
  - **教训**：为了保证 100% 的跨环境编译安全，只要代码中含有对 `React.*` 类型空间的引用，必须在文件头部完整保留 `import React` 的声明。
- **列表渲染中第三方/自定义组件属性约束 (TypeScript Prop Validation)**：在 React 中对自定义列表组件渲染时，如果父组件传入了 `key` 属性，在常规的 JSX 下会被 React 内部消费掉，但在使用 TypeScript 时，如果父组件与自定义组件的 `Props` 接口设计过于严苛，仍然会发生 `Property 'key' does not exist on type...` 的报错。
  - **教训**：在设计通用组件 Props 接口时，若父组件可能传入原生 React 注入属性（如 `key`），可以通过定义 `key?: React.Key` 的声明，在不破坏架构的前提下完美规避编译器的类型噪音。
- **Vite 独有 API (如 import.meta) 的跨环境类型兼容**：Vite 依赖特殊的 `vite/client` 类型来注解 `import.meta.env`，如果项目的 `tsconfig.json` 没有全局声明对应的 `types`，极易因类型缺失引发 lint 失败终止 CI/CD 流程。
  - **教训**：在非强侵入式的前后端联调中，采用 `(import.meta as any).env` 是一种快速、稳健且对代码库干扰最小的“优雅逃生方案”，能确保核心功能和测试验收不受环境阻碍。

### 2026-05-18 经验总结 3
- **ES 模块作用域下使用 `__dirname` 导致后台服务崩溃**：
  - **现象描述**：为了将用户的头像数据以 JSON 格式持久化落盘，在 `server.ts` 中通过 `path.join(__dirname, 'user_avatars.json')` 构造物理存储路径。但在 ES Module 规范下（`"type": "module"`），`__dirname` 变量并未在全局上下文定义，导致 Node 服务器在启动时直接崩溃并抛出 `ReferenceError: __dirname is not defined`。
  - **教训与对策**：在 ES 模块化架构中，若无强烈的绝对路径绑定需求，直接使用相对路径（如 `./user_avatars.json`）是最轻量、免依赖、跨系统环境最安全的方案，能彻底避免 ES Module 对 Node 传统全局变量支持不佳导致的崩溃事件。
